# Spec: M20-PRD — Órdenes de Producción y KDS

**Fecha:** 2026-06-12  
**Estado:** En Revisión  
**Milestone:** M20-PRD  

## 1. Visión General
Este milestone implementa la ejecución en cocina de las Órdenes de Producción (OP). Introduce el *Kitchen Display System* (KDS) para que los operadores gestionen las comandas en tiempo real, el control de tiempos de producción, y el cierre de las órdenes con cálculo automático de varianza (mermas) y movimientos de inventario PEPS.

## 2. Requerimientos Funcionales

### 2.1 Vista KDS (Kitchen Display System)
*   **Header:** Muestra el nombre del Usuario Activo, su rol, la hora actual y la sede activa para dar contexto claro a la sesión de trabajo.
*   **Grid de Tarjetas:** La vista principal de `/admin/production/kds` mostrará las OP en formato de cuadrícula, optimizado para tablets.
*   **Prioridad:** La prioridad de una orden se define al momento de crearla en `/admin/production/orders/new` (añadiremos este selector allí) o puede ser editada por un supervisor.
*   **Ordenamiento:** Las tarjetas se ordenarán primero por `priority` (Urgente > Alta > Normal > Baja) y luego por `scheduled_date` / `created_at`.
*   **Información de Tarjeta:** Mostrará el nombre del producto, cantidad objetivo a producir (con su unidad), tiempo transcurrido desde que se inició, y botón de acción según el estado.
*   **Detalle de Comanda:** Al hacer clic en una tarjeta, se abre un **Modal de Detalle** de gran tamaño que muestra:
    *   La lista exacta de ingredientes necesarios con sus cantidades escaladas.
    *   La línea de tiempo con los pasos de la receta a seguir.
    *   El botón de acción actual (INICIAR o COMPLETAR) en la parte inferior.

### 2.2 Estados de una Orden de Producción
Las órdenes transitan por los siguientes estados:
1.  **Pending:** Recién creada. Botón KDS: "INICIAR". Al iniciar, registra `started_at`.
2.  **In Progress:** En preparación. Botón KDS: "COMPLETAR".
3.  **Completed:** Orden finalizada. Genera lotes y consume inventario.
4.  **Cancelled:** Anulada sin consumir inventario.

### 2.3 Finalización y Varianza de Rendimiento
*   Al hacer clic en "COMPLETAR", se despliega un formulario solicitando la **Cantidad Real Producida** (ej: si se planearon 50 kg, el usuario debe confirmar que salieron 50 kg o ingresar la cantidad real, ej: 48 kg).
*   **Cálculo de Varianza:** `(Cantidad_Real - Cantidad_Esperada) / Cantidad_Esperada * 100`.
*   **Evaluación de Umbral:**
    *   Se consulta el catálogo del artículo para ver si `yield_alert_enabled` es `true`.
    *   Si está habilitado y el valor absoluto de la varianza es mayor a `yield_alert_threshold_pct`:
        *   **Bloqueo Suave (Soft-Block):** Aparece un modal de advertencia indicando que la cantidad está fuera del límite.
        *   El operador puede "Corregir" o "Continuar de todos modos". Si continúa, la orden se marca con `yield_alert_triggered = true` y guarda el porcentaje de varianza para revisión del supervisor.

### 2.4 Movimientos de Inventario (Backend)
Al pasar una orden a `completed`:
1.  **Consumo de Ingredientes (Egresos):** Para cada ingrediente en `production_order_consumptions`, descontar del almacén de origen usando el método PEPS (FIFO). Registrar movimientos tipo `production_consumption`.
2.  **Generación de Producto (Ingreso):** Crear un nuevo registro en `production_lots` con la cantidad real producida. Ingresar el stock al `target_warehouse_id` (o al almacén origen si no hay destino definido). Registrar movimiento tipo `production_yield`.

## 3. Arquitectura de Datos

Las tablas necesarias (`production_orders`, `production_order_consumptions`, `production_lots`) y la columna `yield_alert_triggered` ya fueron creadas en M19. 

*No se requieren nuevas tablas SQL para este milestone.*

## 4. Lógica de Negocio (Backend/API)

*   **`PATCH /production/orders/{id}/status`**: Endpoint para cambiar de pending -> in_progress (registra `started_at`).
*   **`POST /production/orders/{id}/complete`**: Endpoint crítico.
    *   **Payload:** `{ "qty_produced_base": Decimal, "presentation_id": UUID (opcional), "ignore_variance": boolean }`
    *   **Lógica:** 
        1. Verificar varianza si `ignore_variance=False`. Si excede el umbral del artículo, devolver HTTP 409 con detalles de la desviación (para activar el modal en el frontend).
        2. Si la varianza es aceptable o `ignore_variance=True`, proceder con la transacción de base de datos.
        3. Generar egresos PEPS de los ingredientes (referenciando la OP).
        4. Generar ingreso del producto terminado (referenciando la OP y el nuevo `lot_id`).
        5. Actualizar la OP a `completed`.

## 5. Diseño de Interfaz (Frontend)

*   **`/admin/production/orders`**: Convertir el placeholder en una tabla de listado clásico (tipo Kardex) para supervisores.
*   **`/admin/production/kds`**: Nueva pantalla en formato Grid con tarjetas de colores según el estado (Rojo para urgentes, Azul para en progreso, Gris para pendientes).
*   **`OrderCompletionModal`**: Componente de interfaz que maneja el input de cantidad final, la alerta de varianza y la confirmación final.

## 6. Plan de Pruebas (TDD)
1.  **Unit Test (Backend):** Completar una orden con cantidad exacta -> Verifica consumos e ingresos de stock correctos.
2.  **Unit Test (Backend):** Completar orden con varianza excesiva sin flag `ignore_variance` -> Verifica rechazo (HTTP 409).
3.  **Integration Test:** Transición completa: Pending -> In Progress -> Completed (con varianza forzada y flag `ignore_variance=true`).
4.  **Frontend Test:** Renderizado correcto de tarjetas en el KDS y aparición del Modal de Varianza.
